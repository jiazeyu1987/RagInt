# 瑛泰医疗智能能力建设项目立项书（2.0 修改建议）

依据：`doc/BP/1.0/瑛泰医疗智能能力建设项目立项书1.0.md` + `doc/BP/背景.txt`

目标：将 1.0 从“可讨论稿”提升为“可直接走报审链的正式立项书”，更利于总经办与 CFO 快速审阅、也更符合董事长对 AI 的期待。

---

## 1. 报审链视角：把“董事长/总经办/CFO关心的问题”写成可直接回答的结构

建议在执行摘要后新增一节（1–2 页）：

### 1.1 董事长关注点（建议固定 5 问 5 答）
- 我们要建的“智能能力”到底是什么？能沉淀成公司的什么资产？
- 阶段一除了“展厅展示”，能否同时把后续能力建设的底座（知识库、治理、审计）跑起来？
- 如何确保“可控、安全、合规”，不会出现对外不当表述或越权数据泄露？
- 预计 6–12 个月能看到哪些“确定性成果”（不是概念）？
- 如果效果不及预期，最坏情况能否止损、如何体面退出？

### 1.2 CFO 关注点（建议固定 5 问 5 答）
- 本次请批金额是多少、现金流封顶怎么执行、超支怎么控制？
- CAPEX vs OPEX 的会计处理边界是什么（由财务判断，但项目需给出可解释口径）？
- 供应商费用、人力费用、硬件费用（尤其阶段四本地化）分别是多少，如何拆分与验收？
- 运行后年度成本（阶段五 OPEX）是否会变成“被动持续投入义务”？
- ROI 如何解释：至少用“等价 FTE（30–50 万/年）+ 风险规避 + 复制能力”给出可审阅口径（不做岗位替代承诺）。

### 1.3 总经办关注点（建议固定 4 问 4 答）
- 项目边界清不清楚：哪些能做、哪些坚决不做？
- 组织与职责清不清楚：谁对内容负责、谁对风险负责、谁对交付负责？
- 阶段评估怎么做：指标、验收人、止损条件是否明确？
- 对业务负担：业务部门需要投入多少时间与资源？

## 2. “正式立项书”表达规范：把“建议/可选/口径建议”改为“明确承诺与验收条款”

当前 1.0 文中出现较多“建议：/建议补充：/口径建议：”措辞，容易被理解为“尚未定稿”。建议在 2.0：

- 将正文中的“建议补充/建议”改写为“本阶段交付物/本阶段验收标准/本阶段前置条件/本阶段不做事项”。
- 把“可选项/增强项”集中到附录（或成本章节的“可选增强包”），正文只保留本次报审需承诺的范围。
- 对阶段四“本地化部署”从“建议补充”升级为“评估与决策门”：明确输出物（对比方案 + 成本拆分 + 风险评估 + 建议结论），由管理层决定是否进入本地化实施。

## 3. 财务章节 2.0：补齐 CFO 可直接审的“预算拆分表 + 估算依据 + 付款与验收节点”

结合 `背景.txt` 中“财务模型较简单、需细分人力/物料/一次性投入、需区分本地部署软硬件与人工成本”：

### 3.1 新增《预算拆分与测算依据》附录（建议必须有）
- 按阶段 × 成本性质（一次性/持续）× 成本类型（人力/软件/硬件/服务/运维）拆分。
- 本地化部署（阶段四）单列：GPU/存储/网络/机房（或私有云资源）/实施服务/运维人力。
- 每行给“估算依据”：报价单/市场参考/历史项目/供应商方案/内部人力估算。

### 3.2 新增“付款与验收里程碑”表（降低 CFO 风险感）
- 每阶段至少 3 个节点：立项/方案评审 → 试运行验收 → 阶段验收结项。
- 付款建议与验收强绑定：例如 30% 启动 + 40% 试运行通过 + 30% 阶段验收通过（具体比例由采购/财务定）。

### 3.3 新增“年度 OPEX 封顶与增量审批机制”写法（避免后续被动）
- 阶段五年度 OPEX 明确“保底封顶 + 增量审批”，并写清谁审批、什么触发。

## 4. 治理与组织 2.0：把“内容治理”升级为可执行的 RACI 与流程

建议在第五章增加两张表：

### 4.1 RACI（责任矩阵）
- 总经办：范围与风险最终责任、对外口径审批。
- CFO/财务：预算与会计处理原则把关、超支审批。
- 信息化/IT：账号与权限、审计与运维、供应商管理协同。
- 业务部门 owner：内容有效性与更新、输出边界确认。
- 项目实施方（内/外部）：交付与质量、问题响应 SLA。

### 4.2 内容更新流程（含版本号与留痕）
- 提交 → 审核 → 发布 → 抽检 → 复核（明确频次）。
- 明确“制度/流程变更”与“知识库更新”的联动机制，避免知识库长期失真。

## 5. 技术路线 2.0：补一页“架构与选型原则”，让董事长放心、让 IT 可落地

建议增加一个简洁小节（不需要写太多技术细节，但要可审）：

- 架构分层：数据接入/清洗 → 知识库（含权限）→ 检索/推理 → 应用端（展厅、问答、检索）→ 审计与监控。
- 选型原则：可替换（模型可切换）、可审计（引用与日志）、可隔离（权限与数据分区）、可扩展（多部门/子公司）、可本地化（阶段四预留）。
- 关键非功能指标：响应时间、可用性、审计留痕、越权拦截、知识更新时效。

## 6. 风险章节 2.0：把“风险控制”落到清单与机制，而不是描述性语言

在第六章基础上建议新增可直接执行的清单：

- 《数据准入清单（白名单）》与《禁止入库清单（黑名单）》：由总经办/法务/信息安全共同确认，作为上线前置条件。
- 《高风险问题清单》：涉及合规结论、质量结论、对外承诺、合同条款解读等，默认转人工或仅提供引用不作结论。
- 《对外口径机制》：展厅话术与对外宣传材料版本管理、审批链、发布渠道与责任人。

## 7. 指标体系 2.0：把“可验收指标”固化为“每阶段 3–6 个硬指标 + 取数方式”

建议新增“指标定义表”（每项包含：定义、计算口径、取数方式、目标值、责任人）：

- 使用：覆盖人数、周活/月活、核心场景使用占比。
- 质量：有引用回答占比、抽检准确率、转人工率、投诉/纠错闭环时效。
- 效率：检索耗时下降、资料整理周期缩短、培训答疑响应时间。
- 风险：越权拦截次数、审计日志完备率、敏感内容命中与处置时效。

## 8. 范围控制 2.0：把阶段三/四的“扩展”写成门槛和节奏，防止失控

- 阶段三明确“首批部门数量上限 + 首批上线场景上限 + 每批次评估周期”，并写入“未达标不扩容”的规则。
- 阶段四明确“子公司接入条件”：制度齐备、owner 到位、权限体系可对接、运维责任明确。

## 9. 文档级修订建议（细节但影响观感）

- 全文统一使用正式措辞，减少括号内口语化说明；将“建议”类表达转为“本项目承诺/约束/验收”。
- 统一章节编号与表格编号；将关键表格（预算拆分、付款验收、指标定义、RACI）集中在目录中可快速定位。
- 增加“名词解释/缩略语”小节（AI、RAG、TTS、OPEX/CAPEX 等），便于非技术评审阅读。
