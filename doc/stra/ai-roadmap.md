# 公司 AI 化推进路线（PLM/ERP/MDS 未完全打通前提）

## 1. 背景与原则

现状：PLM、ERP、MDS（以及 MES/CRM/售后等）之间的数据流与主数据还未完全贯通。此阶段若直接全面上 AI，最容易出现两类问题：**答案不可信**（数据不一致/滞后）与 **无法规模化**（每个场景单点对接、难复用）。

推进原则（建议作为总纲）：
- 先“数据可用/可追溯”，再“模型更聪明”。
- 先“高频、低风险、可量化收益”场景，再进入“强流程、强权限、强责任”的核心业务环节。
- 以“平台能力 + 治理机制 + 可复用资产（数据/知识/接口）”为交付物，而不是只交付 Demo。
- 所有 AI 输出都要有“出处/证据/责任边界”（尤其是 PLM/ERP 相关）。

## 2. 分阶段总览（触发契机 → 要做什么 → 交付什么）

建议分 5 个阶段推进（可并行，但不要跳过前置条件）。

| 阶段 | 触发契机（进入条件） | 主要工作（做什么） | 关键交付物（交付什么） |
|---|---|---|---|
| P0 启动与盘点（0~4周） | 管理层明确 AI 目标与预算；指定牵头人 | 盘点系统/数据/权限/场景；确定 3~5 个优先场景；定义指标 | `AI 目标与范围说明`、`场景优先级清单`、`数据资产与接口清单`、`指标口径与验收方式` |
| P1 数据可用化（1~3个月） | 找到“一个主数据域”可率先治理（例如物料主数据） | MDS 定义与对齐；建立跨系统 ID 映射；打通最小数据链路；数据质量度量 | `主数据字典/编码规范`、`跨系统映射表`、`数据质量报表`、`最小数据通道（ETL/API）` |
| P2 AI 就绪平台（2~4个月，可与P1并行） | P1 有稳定数据源/接口；安全/合规机制明确 | 建 RAG/知识库平台；统一权限/审计；统一 Prompt/工具调用；观测与成本管控 | `企业知识库与索引规范`、`权限模型与审计日志`、`模型网关/调用封装`、`AI 可观测面板（质量/成本/时延）` |
| P3 流程内 AI（3~6个月） | P1+P2 稳定运行；关键数据可信；责任边界明确 | 把 AI 嵌入到 PLM/ERP/MES 的关键节点；人机协同审批；闭环反馈 | `流程节点清单与改造方案`、`人机协同SOP`、`审批与回写机制`、`闭环数据（反馈标签）` |
| P4 全面规模化（6~12个月） | 多部门可复用平台；收益可量化；治理成熟 | 复制到更多域；形成 AI COE；持续运营与优化 | `AI 资产目录`、`COE 运作机制`、`年度迭代路线图`、`跨域复用模板/SDK` |

> 注：若某阶段推进阻力大，通常不是“模型不行”，而是“数据/权限/流程/责任”尚未准备好。

## 3. 各阶段详细拆解

### P0 启动与盘点（0~4周）

触发契机（进入条件）：
- 总经办明确：AI 的业务目标（降本/提效/增收/风险控制）与预算上限；
- 明确牵头组织（建议：信息化 + 业务条线 + 合规/法务），并指定产品负责人/项目经理；
- 允许跨部门拿到“必要的只读数据”做盘点。

需要做什么：
- 系统盘点：PLM/ERP/MDS/MES/CRM/售后/知识库/文件盘等，列出数据域、接口方式、权限现状、数据更新频率；
- 场景盘点：收集 20~50 个候选场景，按“收益/风险/数据可得/落地周期”打分，选出 3~5 个优先场景；
- 指标定义：每个场景定义可量化指标（时长、错误率、返工率、交付周期、库存周转、客诉响应等）；
- 风险清单：敏感数据范围、合规约束、可能的误用方式与防控策略。

交付物：
- `场景优先级清单（含评分与理由）`
- `数据资产清单（数据域/系统/字段/负责人/更新频率）`
- `接口清单（已有API/ETL/导出方式、打通难度）`
- `指标口径与验收方案（怎么测、测多久、谁签字）`

### P1 数据可用化（1~3个月）

触发契机（进入条件）：
- 至少选定一个“主数据域”作为突破口（建议从物料/客户/供应商/设备中选一个）；
- 业务能接受主数据标准化（编码/字段含义/归属）；
- 允许建立跨系统映射与最小同步机制。

需要做什么：
- 主数据治理：字段定义、编码规范、生命周期、责任人（Owner）；
- 跨系统 ID 对齐：PLM 物料号/ERP 物料号/MDS 主键映射，建立“黄金记录”；
- 最小数据通道：先打通只读与增量更新（不追求一次性全量打通）；
- 数据质量：建立规则（唯一性、完整性、一致性、及时性），并持续出报表；
- 数据可追溯：每条记录能追到来源系统与更新时间。

交付物：
- `主数据字典与编码规范`
- `跨系统映射表（含冲突处理规则）`
- `数据质量仪表盘（DQ）与告警规则`
- `最小数据同步机制（ETL/CDC/API）与运行手册`

### P2 AI 就绪平台（2~4个月，可与 P1 并行）

触发契机（进入条件）：
- P1 至少有一个稳定可信的数据域可用（哪怕范围不大）；
- 有明确的权限/审计要求（谁能问、能看到什么、能导出什么）。

需要做什么：
- 企业知识库建设：把散落在文档/邮件/规范/FAQ 的知识结构化（RAG），并与主数据做关联；
- 数据清洗与入库工具链：抽取、去噪、去重、分段、元数据（来源/版本/敏感级别）；
- 权限与审计：角色权限（管理员/运营/只读访问），密钥轮换、操作留痕；
- 模型网关：统一接入大模型/向量库/ASR/TTS 等，集中做限流、成本统计、日志脱敏；
- 可观测：质量（命中率/引用率/人工采纳率）、时延、失败率、成本（token、向量化、存储）。

交付物：
- `知识库平台（内网可部署）与数据集规范`
- `清洗与导入工具链（脚本/流程/模板）`
- `权限模型、审计策略、密钥轮换SOP`
- `模型网关/SDK（统一调用封装）`
- `AI 可观测面板（质量/成本/时延/失败原因）`

### P3 流程内 AI（3~6个月）

触发契机（进入条件）：
- P1 主数据在关键流程中可信（至少一个流程闭环）；
- P2 的权限/审计/成本控制跑通；
- 能明确“谁对 AI 建议负责，谁最终审批”。

需要做什么：
- 选 2~3 个“流程节点”落地（示例）：PLM 变更评审、ERP 采购申请校验、MDS 主数据新增校验、质量异常原因归因；
- 人机协同：AI 只给建议 + 证据，最终由人审批；审批结果反哺模型与规则；
- 回写机制：通过 API/任务单把“建议/审批/结果”写回系统（形成闭环数据）；
- 风控：敏感字段遮蔽、越权访问拦截、关键操作双人复核。

交付物：
- `流程节点改造方案（含责任边界）`
- `人机协同SOP（审批/回写/复盘）`
- `闭环数据结构（反馈标签、采纳率、错误归因）`
- `权限与风控策略（字段级、角色级、审计）`

### P4 全面规模化（6~12个月）

触发契机（进入条件）：
- P3 至少 1~2 个流程证明 ROI（可量化），并且稳定运行；
- 平台能力可复用（新场景接入成本显著下降）；
- 具备长期运营团队与机制。

需要做什么：
- 复制到更多数据域（客户/供应商/设备/工艺/质量）；
- 建立 AI COE（中心化能力与资产复用）：标准、模板、评测、治理；
- 建立持续迭代：每月/每季度的模型与知识库更新节奏；
- 建立“成本-收益”经营视角：token 成本、平台成本与节省的人力/缩短的周期对比。

交付物：
- `AI 资产目录（模型、Prompt、工具、数据集、评测集）`
- `COE 运作机制（流程、权限、评审、上线标准）`
- `年度路线图与预算模型`
- `跨域复用 SDK/模板（接入清单、评测模板、监控模板）`

## 4. 触发契机的常用信号（建议用于决策）

以下信号出现，通常意味着“可以进入下一阶段”或“必须先补课”：
- **进入 P1 的信号**：业务愿意统一编码与字段口径；系统间能建立主键映射；有数据 Owner。
- **进入 P2 的信号**：已有稳定可用数据源；能定义权限边界；能接受审计与成本统计。
- **进入 P3 的信号**：一个关键流程的数据一致性提升到可用；能定义审批责任；能做回写闭环。
- **进入 P4 的信号**：至少一个流程 ROI 证明；接入新场景成本下降；治理机制能自循环。

## 5. 建议从哪些场景先启动（不依赖全打通）

在 PLM/ERP/MDS 未完全打通前，优先选“只读、低风险、强价值”的场景：
- 规范/制度/流程问答（带引用）
- 物料/图纸/变更查询（只读）与一致性检查（提示冲突不自动改）
- 采购/质量/售后 FAQ 与相似案例检索
- 报表解释与异常定位（只读，输出依据与口径）

后续再进入“写入/审批/强责任”的流程内 AI。
